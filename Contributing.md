# 🚀 Contributing to PySpecTrace
Thank you for your interest in making PySpecTrace better! Whether you are a spectroscopist fixing a bug or a developer adding a new real-time analysis feature, we value your help.

## 🌟 Ways to Contribute

**Feature Requests**: Suggest new tools for the UV-VIS-NIR real-time analysis suite.

**Bug Reports**: Help us find and fix edge cases in spectrometer interfacing.

**Code Contributions**: Improve our processing algorithms or GUI responsiveness.

**Documentation**: Fix typos or add examples to our user guides.

## 🛠️ The Contribution Process

Follow these steps to get your changes into the main project:

1. **Fork the Repo**: Fork the development branch of our repository to your own GitHub account.

2. **Branch Out**: Create a new working branch from your fork to start your changes.

3. **Code & Improve**: Make your modifications. Whether it's a small fix or a major new UI component, we appreciate the effort!

4. **Meaningful Commits**: Commit your changes when you’re happy with them. Please use clear, descriptive commit messages (e.g., feat: add real-time peak detection instead of fix: stuff).

5. **Submit a PR**: When finished, create a Pull Request (PR) back to our development branch.

- Note: If you are solving an existing bug, please link your PR to that Issue.

6. **Review & Merge**: The development team will review your PR. We might suggest small changes to ensure it aligns with the core architecture. You can apply these through the GitHub UI or by pushing new commits to your branch.

- Success! Once approved, your PR will be merged. 🎉

## 📝 Contribution Guidelines

**Environment**: Ensure you have the necessary spectral processing libraries installed. We recommend testing your GUI changes against the simulated spectrometer mode first.

**Code Style**: We aim for clean, readable Python code.

**Be Kind**: Please follow our [Code of Conduct]—we are a community of researchers and developers supporting each other.

## 🏅 Recognition

Significant contributors will be acknowledged in our documentation and, where appropriate, invited to be co-authors on future software publications (e.g., SoftwareX or JORS).
